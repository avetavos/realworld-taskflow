## What we're building

`password.rs` มีสองฟังก์ชัน: `hash_password(plain: &str) -> anyhow::Result<String>` ที่แปลงรหัสผ่าน plaintext ให้เป็น hash string แบบ Argon2 ที่ปลอดภัยพอจะเก็บใน column `users.password_hash` และ `verify_password(plain: &str, hash: &str) -> bool` ที่เช็ครหัสผ่าน plaintext ตอน login เทียบกับ hash ที่เก็บไว้ โดยไม่ต้อง reverse มันกลับมาเลย

ทุกส่วนที่เหลือใน Module 4 ต่อยอดจากสองฟังก์ชันนี้: `register` เรียก `hash_password` ก่อน `INSERT` และ `login` เรียก `verify_password` เทียบกับแถวที่เพิ่ง fetch มา ทั้งสองฟังก์ชันไม่คุยกับฐานข้อมูล, Redis, หรือ Axum เลย — มันเป็น building block ที่บริสุทธิ์และ test ง่าย

## Why

รหัสผ่านคือความลับหนึ่งเดียวในทั้งระบบที่ผู้ใช้เลือกเอง ใช้ซ้ำข้ามบริการอื่น และไม่คาดหวังเลยว่าการรั่วของฐานข้อมูล *ของคุณ* จะไปเปิดเผยมันได้ ถ้า `users.password_hash` รั่วออกไปเมื่อไหร่ — backup ที่วางผิด S3 bucket, replica ที่ถูกเจาะ, SQL injection — ผลลัพธ์ที่ยอมรับได้มีอยู่ทางเดียวคือข้อมูลที่รั่วออกไปนั้น *ไร้ประโยชน์* สำหรับคนที่เอาไป นั่นคือสิ่งที่ password hashing function รับประกันได้ แต่การเก็บ plaintext หรือ encryption แบบย้อนกลับได้ทำไม่ได้

**Hashing แบบทางเดียว ต่างจาก encryption** Encryption ถูกออกแบบให้ reverse ได้ — ใครก็ตามที่ถือ key สามารถกู้ plaintext กลับมาได้ ซึ่งหมายความว่า key กลายเป็น single point of failure และเป็นเป้าหมายชิ้นโตชิ้นเดียว Hashing ทิ้ง plaintext ไปเลยทั้งหมด `hash_password` สร้าง string ขนาดคงที่ที่ไม่มี `decrypt` ให้ การ verify การ login คือการ hash *ค่าที่พยายาม login* ด้วยวิธีเดียวกันแล้วเทียบ — ไม่เคย decrypt ค่าที่เก็บไว้เลย

**Hash function ทั่วไป (SHA-256, MD5) ก็ยังผิดสำหรับรหัสผ่าน** แม้จะเป็นทางเดียวก็ตาม hash function ทั่วไปถูกสร้างมาให้ *เร็ว* — นั่นคือคุณสมบัติที่คุณต้องการสำหรับ checksum พอดี และเป็นคุณสมบัติที่คุณไม่ต้องการเลยสำหรับรหัสผ่าน เพราะมันคือสิ่งเดียวกันที่ทำให้การ brute-force ทุกรหัสผ่านที่เป็นไปได้เร็วสำหรับผู้โจมตีที่มี GPU ด้วย Argon2 คือ **password hashing function**: จงใจให้ช้าและกิน memory มาก ปรับจูนมาให้การ hash รหัสผ่านหนึ่งตัวใช้เวลาเสี้ยววินาทีสำหรับ server ของคุณ แต่ทำให้การ hash การเดารหัสหลายพันล้านครั้งแพงเกินไปสำหรับผู้โจมตี

## Pros & cons

**Argon2 (ตัวที่เราใช้) เทียบกับ bcrypt**

- ข้อดี: Argon2 เป็น memory-hard — มันมีค่าใช้จ่าย RAM ที่ปรับได้ต่อการ hash หนึ่งครั้ง ไม่ใช่แค่เวลา CPU เท่านั้น ซึ่งเจาะจงเอาชนะ rig สำหรับ crack ด้วย GPU และ ASIC ที่ได้เปรียบด้านความเร็วจากการขนานจำนวนมากเหนือ memory ราคาถูก มันยังเป็นผู้ชนะการแข่งขัน Password Hashing Competition ปี 2015 และเป็น algorithm ที่ OWASP แนะนำอย่างจริงจังสำหรับระบบใหม่ `argon2 = "0.5"` ให้เรา `Argon2::default()` — Argon2id พร้อม parameter ที่ OWASP แนะนำ — โดยไม่ต้อง tune เองเลย
- ข้อเสีย: bcrypt มีประวัติการใช้งานจริงยาวนานกว่า (ใช้กันอย่างแพร่หลายตั้งแต่ปี 1999) และมี API ที่เรียบง่าย ตั้งค่าผิดยากกว่าในบาง ecosystem bcrypt ยัง truncate input ที่เกิน 72 byte แบบเงียบ ๆ ซึ่ง Argon2 ไม่ทำ — เป็น footgun ที่น้อยกว่าหนึ่งอัน แต่ไม่มากพอที่จะยอมสละความ memory-hard ของ Argon2 สำหรับระบบใหม่ในปี 2026

**รูปแบบ PHC string — hash พก parameter ของตัวเองไปด้วย (ตัวที่เราใช้) เทียบกับเก็บ salt/parameter แยก column**

- ข้อดี: `Argon2::default().hash_password(...)` คืน string ที่อธิบายตัวเองได้อย่าง `$argon2id$v=19$m=19456,t=2,p=1$<salt>$<hash>` — algorithm, version, ค่าใช้จ่ายด้าน memory/time/parallelism, salt และ hash อยู่ใน string เดียวนั้นทั้งหมด `PasswordHash::new(hash)` parse มันกลับออกมาได้ ดังนั้น `verify_password` จะใช้ parameter เดิมเป๊ะที่รหัสผ่านถูก hash ไว้ตอนแรกเสมอ แม้เราจะปรับ cost parameter ของ `Argon2` สำหรับ hash *ใหม่* ในภายหลังก็ตาม `users` ต้องการแค่ column `text` เดียว (`password_hash`)
- ข้อเสีย: ถ้าเราอยากจะ query "ผู้ใช้คนไหนถูก hash ด้วย parameter ที่ล้าสมัยแล้ว" เพื่อทำ rehash migration นั่นหมายถึงต้อง parse PHC string ทีละแถว แทนที่จะเป็น SQL ธรรมดา `WHERE cost_param < N` เป็น trade-off ที่สมเหตุสมผลสำหรับโปรเจกต์ขนาดคอร์สแบบนี้ ระบบขนาดใหญ่อาจแยก cost parameter ออกมาเป็น column ของตัวเอง

**salt แบบสุ่มต่อรหัสผ่านหนึ่งตัว (ตัวที่เราใช้ ผ่าน `SaltString::generate`) เทียบกับไม่มี salt / salt ที่ใช้ร่วมกัน**

- ข้อดี: ผู้ใช้สองคนที่มีรหัสผ่าน `hunter42` เหมือนกันเป๊ะ จะได้ค่า `password_hash` ที่ต่างกันโดยสิ้นเชิง เพราะแต่ละ hash ฝัง salt สุ่มของตัวเองไว้ นี่เอาชนะ rainbow table (การเก็บ hash ไว้ล่วงหน้าเพื่อค้นหา) และหยุดผู้โจมตีที่ crack รหัสผ่านของผู้ใช้คนหนึ่งได้จากการได้ฟรีกับทุกคนที่ใช้รหัสผ่านซ้ำกัน
- ข้อเสีย: ไม่มีอะไรที่ควรพูดถึง — hash ที่ไม่มี salt หรือใช้ salt ร่วมกันคือช่องโหว่ตรง ๆ ไม่ใช่ trade-off `SaltString::generate(&mut OsRng)` ไม่มีต้นทุนเพิ่มเติมเลยในการเรียกให้ถูกต้อง

## Build it

### 1. เพิ่ม dependency `argon2`

```bash
cd taskflow/backend
cargo add argon2 -p api
```

คำสั่งนี้เพิ่ม `argon2 = "0.5"` ใต้ `[dependencies]` ใน `api/Cargo.toml`

### 2. `password.rs`

สร้าง `taskflow/backend/api/src/auth/password.rs`:

```rust
use argon2::{
    password_hash::{rand_core::OsRng, PasswordHash, PasswordHasher, PasswordVerifier, SaltString},
    Argon2,
};

pub fn hash_password(plain: &str) -> anyhow::Result<String> {
    let salt = SaltString::generate(&mut OsRng);

    let hash = Argon2::default()
        .hash_password(plain.as_bytes(), &salt)
        .map_err(|e| anyhow::anyhow!("failed to hash password: {e}"))?
        .to_string();

    Ok(hash)
}

pub fn verify_password(plain: &str, hash: &str) -> bool {
    let Ok(parsed_hash) = PasswordHash::new(hash) else {
        return false;
    };

    Argon2::default()
        .verify_password(plain.as_bytes(), &parsed_hash)
        .is_ok()
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn hash_and_verify_round_trip() {
        let hash = hash_password("correct horse battery staple").unwrap();
        assert!(verify_password("correct horse battery staple", &hash));
        assert!(!verify_password("wrong password", &hash));
    }

    #[test]
    fn rejects_a_malformed_hash() {
        assert!(!verify_password("anything", "not-a-phc-string"));
    }
}
```

มีรายละเอียดสองสามอย่างที่ควรพูดถึง:

- `SaltString::generate(&mut OsRng)` ใช้ตัวสร้างเลขสุ่มที่ปลอดภัยเชิง cryptographic ของระบบปฏิบัติการ — ไม่เคยใช้ salt ที่คาดเดาได้หรือคงที่เลย
- `hash_password` คืนค่า `anyhow::Result<String>` ตรงกับ pattern ของทุกฟังก์ชันที่ fail ได้และไม่ใช่ HTTP ตัวอื่นที่เราเขียนมาตั้งแต่ [config-tracing](/taskflow/th/backend-foundations/config-tracing/) ประเภท error ของ `argon2::password_hash::Error` ไม่ได้ implement `std::error::Error` ในแบบที่ `?` จะแปลงอัตโนมัติได้ เราจึงห่อมันด้วย `anyhow::anyhow!` พร้อมข้อความสั้น ๆ
- `verify_password` คืนค่า `bool` ธรรมดา ไม่ใช่ `Result` — ทั้ง hash ที่เก็บไว้แบบผิดรูปและรหัสผ่านที่ผิดจริง ๆ ต่างก็แค่หมายถึง "ไม่ผ่านการยืนยันตัวตน" pattern `let ... else { return false }` ทำให้สิ่งนี้ชัดเจนแทนที่จะบังคับให้ทุกจุดเรียกต้องจัดการ parse error แยกกัน
- module `#[cfg(test)]` ยืนยันว่า round trip ทำงาน และ input ที่เป็นขยะจะ fail แบบปิด (คืน `false`) แทนที่จะ panic

### 3. เริ่ม module `auth`

สร้าง `taskflow/backend/api/src/auth/mod.rs`:

```rust
pub mod password;
```

เพิ่ม `mod auth;` ใน `taskflow/backend/api/src/main.rs`:

```rust
mod auth;
mod config;
mod db;
mod error;
mod state;
```

`auth` คือ directory module — `auth/mod.rs` คือจุดเริ่มต้นของมัน และทุกบทเรียนใน module นี้จะเพิ่มบรรทัด `pub mod ...;` อีกหนึ่งบรรทัดเข้าไป ขณะที่เราสร้าง `jwt.rs`, `middleware.rs` และ `handlers.rs`

## Verify

```bash
cargo check -p api
```

จากนั้นรัน unit test ใหม่:

```bash
cargo test -p api auth::password
```

ผลลัพธ์ที่ควรได้:

```
running 2 tests
test auth::password::tests::rejects_a_malformed_hash ... ok
test auth::password::tests::hash_and_verify_round_trip ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

`cargo check -p api` จะ compile ผ่านพร้อม warning ประเภท `dead_code`/`never constructed` อยู่บ้างในตอนนี้ — `password.rs` ยังไม่ถูกเรียกจาก handler ใดจนกว่าจะถึง [handlers](/taskflow/th/auth/handlers/) ที่เชื่อม `register` และ `login` เข้ามา นี่เป็นเรื่องปกติในขั้นนี้ เหมือนกับทุก module ที่เป็นโครงเตรียมไว้ก่อนหน้านี้

## Recap

คุณสร้าง `hash_password` และ `verify_password` ใน `password.rs` โดยใช้ Argon2id พร้อม salt สุ่มต่อรหัสผ่านหนึ่งตัวผ่าน `SaltString::generate` รูปแบบ PHC string หมายความว่า `password_hash` เป็น column `text` เดียวที่พก parameter ของ algorithm ไปด้วยในตัวเอง ดังนั้น `verify_password` จะเช็คเทียบกับค่าที่ hash ถูกสร้างขึ้นเป๊ะเสมอ คุณยังเริ่ม directory module `auth` — `auth/mod.rs` — ที่ Module 4 ที่เหลือจะต่อยอดไป ต่อไปเราจะออกและ verify signed session token ใน [jwt](/taskflow/th/auth/jwt/)
